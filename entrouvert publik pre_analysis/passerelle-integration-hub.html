Passerelle (Integration Hub)

Summary & Integration

Passerelle is Publik’s middleware hub, responsible for connecting the platform to external information systems and APIs[33]. It acts as a uniform web services layer: external business applications (e.g. a city’s ERP, GIS, or national data service) are integrated by writing connectors to Passerelle, and the other Publik modules then talk to Passerelle rather than each system individually. In the ecosystem, Passerelle is invoked by modules like WCS (during form display or workflow actions) or Combo (for portal cells) whenever data from an outside system is needed. For instance, if a form needs to populate a dropdown with data from a national address API or if a workflow needs to send a request to an external CRM, Passerelle handles that request. It essentially serves as a webservice hub that speaks multiple protocols and formats so that Publik’s internal components can remain agnostic of external peculiarities[33]. Integration-wise, Passerelle has many sub-modules (connectors) for specific third-party services – the Entr’ouvert team maintains connectors for various government APIs (like address lookup, company info) and for vendor software used by local governments[34]. These connectors plug into Passerelle, and WCS or other modules simply call Passerelle’s unified API to trigger them. By translating data formats and protocols, Passerelle ensures that, for example, a SOAP XML service or a SQL database can be accessed and return JSON to Publik’s front-end.

Philosophy, Design & Usage

Passerelle’s philosophy is to provide interoperability through abstraction. Rather than hard-coding integrations in each module, a single integration layer simplifies maintenance and encourages reusability of connectors across different forms or portals. It is designed to handle as many data exchange patterns as possible: it natively understands JSON, XML, CSV, and even direct SQL, and can handle REST or SOAP webservice protocols[35]. The design is likely based on Django (for the web framework and admin interface), given Entr’ouvert’s general use of Python/Django. Connectors in Passerelle are probably implemented as Django apps or classes that define how to interact with a specific service (with configuration for endpoints, authentication, etc.). The choice to include support for multiple formats/protocols reflects a pragmatic approach – local governments have diverse systems, so Passerelle must be a Swiss-army knife. In usage, administrators or developers configure connectors usually by installing the appropriate Python package and adding an instance via Passerelle’s admin (for example, configuring the credentials for a particular API). Then WCS form designers can use those connectors by referencing them in form fields or workflow actions, without worrying about the underlying details. The philosophy of openness is evident: Passerelle only works with open, documented, structured protocols, meaning proprietary or closed systems need to offer some standard interface for Passerelle to use[35]. This encourages the adoption of open standards in the e-government context. Additionally, usage of Passerelle encourages decoupling – if a city changes a backend system, updating or replacing a connector in Passerelle should suffice, rather than altering all forms/workflows that use that data.

Key Features & Functions

The core function of Passerelle is to serve as a translator and orchestrator for data exchanges. It provides a uniform RESTful API to Publik’s other modules while internally handling whatever protocol the target system requires[36]. Key features include: a library of pre-built connectors for a wide range of systems (from national government APIs to specific vendor software)[34]; support for various data formats (JSON, XML, CSV, SQL queries) as input/output[36]; and probably a plugin mechanism to add new connectors easily. Passerelle likely offers logging and monitoring of calls (so one can trace what calls were made for auditing). Another function is data transformation – many connectors will need to transform data fields between Publik’s format and the external system’s format (for instance, mapping field names or encoding values). Passerelle provides a framework to do these transformations systematically. It also can handle authentication to external services (storing API keys, tokens, certificates as needed for each connector). In terms of essential operations, from the perspective of WCS or Combo, using Passerelle looks like calling a webservice: e.g., WCS might execute an action “call webservice X with these parameters,” and Passerelle’s job is to route that to the right connector, handle the request (perhaps converting to a SOAP XML envelope or SQL query), then parse the response and give it back in a normalized format. Passerelle speaks common web protocols: for example, it can consume REST APIs and return JSON (likely the native format it uses with Publik modules is JSON), or it can act as a SOAP client as needed[36]. It is essentially a hub-and-spoke design: multiple external “spokes” connect to the central hub (Passerelle), simplifying the design of the rest of Publik. Its initial essential function set included those fundamental connectors and the ability to define new ones; given the version number (8.x) and numerous sub-packages listed (e.g., passerelle-montpellier-sig, passerelle-grandlyon-elyx, etc. in the package repository[37][38]), a lot of real-world connector logic is encapsulated in these modules.

Unknowns or Gaps

While we know Passerelle’s broad capabilities, the internal architecture is not fully detailed in public docs. It’s not explicitly stated, for example, how one writes a new connector – presumably by subclassing some base connector class in Python, but the specifics are not in the documentation we’ve seen. Also unknown is how Passerelle is deployed in a multi-tenant environment: does each tenant have its own instance of Passerelle (with isolated connector configurations), or does one Passerelle serve multiple tenants with tenant-aware connectors? Given Hobo’s multitenancy, it could be the latter, but documentation is lacking. Another gap is how synchronous vs asynchronous calls are handled. If a connector call is long-running (say a slow SOAP service), does Passerelle offer asynchronous execution or timeouts? The behavior in such scenarios isn’t described publicly. Additionally, error handling strategy (e.g., if an external service is down, how are errors propagated to WCS workflows) is not explicitly documented – likely it surfaces an error status, but specifics would be known by examining code or manuals. The extent of data caching is also unclear (possibly none, leaving caching to each service or to calling modules, to ensure real-time data). Lastly, security considerations (like how credentials are stored, how access to connectors is controlled) are not discussed in the high-level sources, leaving a bit of guesswork for implementers until they see configuration examples.

Suppositions

We can suppose that Passerelle’s development will continue to expand its library of connectors as new needs arise (for example, if new national APIs are released, connectors will be added). There might also be an ongoing effort to standardize connectors – for instance, if many cities use similar systems, Entr’ouvert likely refactors code to reuse connector components. Another supposition is that performance optimizations could be a focus: as more processes rely on Passerelle, the team might implement caching for certain read-only data (like caching a list of streets for a short period to avoid frequent external calls) while ensuring data freshness where it counts. It’s also reasonable to guess that Passerelle uses Django Rest Framework or a similar concept under the hood to expose its own API, and that future versions might formalize a connector SDK for third-party developers to write their own plugins more easily. In absence of explicit documentation, one assumes each connector is packaged separately (as seen by many python3-passerelle-... packages[39]) and that they are maintained alongside core Passerelle. Given the emphasis on open standards, a safe supposition is that if an external system does not provide a proper API, it won’t be directly supported until it does – thus pushing vendors to open up. For new developers, it’s reasonable to think that learning Passerelle involves studying existing connector code to mimic the patterns. Finally, as Publik installations grow, one might foresee Passerelle being scaled horizontally or separated out if needed (for example, heavy integrations could be offloaded to separate workers), though at present it’s likely integrated in the same web stack. Overall, Passerelle’s trajectory will likely involve deeper integration capabilities (possibly supporting new protocols or data formats) and keeping up with the evolving landscape of e-government web services.
