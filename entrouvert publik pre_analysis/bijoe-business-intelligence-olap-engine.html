BiJoe (Business Intelligence / OLAP Engine)

Summary & Integration

BiJoe is Publik’s business intelligence module, essentially a BI/OLAP engine that works with data collected by other Publik components[50]. The name “BIJoe” implies BI (Business Intelligence) with a bit of whimsy. It serves to simplify querying a PostgreSQL database organized in a star schema (a typical data warehouse schema) and to produce dashboards or reports for analytical purposes[50]. In the Publik ecosystem, BiJoe sits downstream of the operational databases: for example, WCS holds transactional data (forms submitted, requests processed), and BiJoe can take that data (through an ETL or export process) into a separate schema optimized for analysis (facts and dimensions). The integration is such that WCS (and possibly Chrono, Lingo, etc.) push or export data to BiJoe’s schema via the WCS-OLAP extension or similar tools. Then BiJoe provides a query interface or API to retrieve aggregated metrics. BiJoe can integrate with the Combo portal by powering analytics dashboards for administrators – perhaps graphs or stats on the agent back-office. It might also be used in the Publik Famille context to analyze usage statistics (like number of child registrations, payments processed, etc.). Importantly, BiJoe is a library as well, meaning developers can use it within Django to run OLAP queries without writing complex SQL[50]. In summary, BiJoe acts as the analytics layer, ensuring decision-makers can get insights from the data in Publik without needing an external BI tool.

Philosophy, Design & Usage

The philosophy of BiJoe is to leverage the power of SQL (particularly PostgreSQL) for analytics, but hide the complexity from the user. By using a star schema (fact and dimension tables), it organizes data in a way optimal for OLAP queries (like grouping, summing, etc.), and by providing a Django app interface, it allows generating these queries through simpler Python code or UI selections. The design suggests that BiJoe likely defines models corresponding to dimensions (e.g., date, form type, location) and fact tables (e.g., submissions fact with measures like count, processing time). The mention that it’s a library and a Django application[50] means it can be embedded in a Django project (like the Publik admin site) to run queries, and also possibly has its own UI (maybe a simple interface to execute predefined queries or to connect to external dashboard tools). BiJoe probably uses Django’s ORM minimally, because OLAP queries often use raw SQL for performance; so the library part might construct SQL queries (for example, to pivot data). In usage, an administrator or developer defines what metrics they want – for instance, “number of requests per month per department” – and BiJoe helps by providing the means to query the star schema instead of the normalized transactional schema. BiJoe’s approach is likely data-agnostic regarding Publik modules: it doesn’t inherently know about forms or appointments, but through extensions like WCS-OLAP, it gets the data structure. The design choice to separate BIJoe indicates a read-only analytic database philosophy: keep heavy analytic queries off the live transactional DB and instead use a dedicated schema or DB for reporting. This improves performance and safety. Usage may involve scheduled jobs to refresh the BI data (for example, nightly ETL from WCS to BIJoe). On the front-end, usage could be through charts or tables presented to managers (like number of requests this week, average processing time, etc.), likely embedded in the agent portal or delivered via reports.

Key Features & Functions

BiJoe’s key functionality is to query data warehouses easily. It provides abstractions so that a developer or advanced user can define a star schema and then run multi-dimensional queries without hand-writing complex SQL with joins and group by. For example, BiJoe might let you retrieve a JSON or Python object of results by specifying dimensions and measures in Python. It likely has functions to define cubes and dimensions. It might use an existing OLAP library or its own querying mechanism. The description highlights producing dashboards, so one feature could be integration with a charting library or at least the ability to output query results that can feed a visualization. Another function is filtering – easily filtering on dimensions (e.g., only data from a certain date range or category) is important in OLAP, and BiJoe would offer a way to apply filters without writing SQL syntax manually. Since it’s a Django app, BiJoe might also include models to configure some default analyses or to store queries. Possibly it offers a simple UI for selecting a dataset and breakdown (though this is speculative; it might rely on code or external tools instead). In essence, its initial essential function was likely to take a star schema in Postgres and allow queries like “give me count of facts grouped by this dimension and that dimension,” returning results quickly. It may also manage connections to the database and ensure that queries are optimized (perhaps using database views or materialized views for performance). If any part of Publik needed to embed a chart, BiJoe would provide the data side of that. Additionally, WCS-OLAP extension presumably works with BiJoe to define the star schema for WCS data (like each form submission becomes a fact with dimensions like form type, submission date, status, etc.), meaning BiJoe’s functionality includes being extensible via such plugins for different data sources.

Unknowns or Gaps

Much of BiJoe’s inner workings are not detailed publicly. We don’t know the exact interface it provides – for instance, is there a DSL (Domain Specific Language) to define OLAP queries in Python? Or does it rely on writing YAML/JSON configs for cubes? Also unknown is how interactive it is – can non-technical users use BiJoe via a GUI, or is it mostly a developer’s tool that outputs data which is then manually turned into charts? The documentation simply notes it simplifies querying and producing dashboards, but not how. The integration with front-end charts is unclear: does BiJoe come with any pre-built dashboard components or does it expect integration with something like Grafana or custom code? Another gap is to what extent BiJoe handles incremental updates. If new data comes in (new form submissions), do the star schema tables update in real-time, or only on demand, or via scheduled tasks? Possibly WCS-OLAP triggers updates, but specifics are not known. Additionally, BiJoe’s scalability is a question – OLAP queries can be heavy; how does BiJoe cope with large data volumes (millions of records)? No info on performance tuning or limitations is readily available. For new developers, the lack of explicit examples in documentation might be a challenge – one would have to read BiJoe’s code or find sample usage to truly grasp how to formulate queries. Moreover, BiJoe is specialized for Postgres (it mentions Postgresql specifically[51]); it’s unknown if it could work with other databases (likely not needed, since Publik standardizes on Postgres). Finally, the origin of the name “BIJoe” is whimsical and the rationale isn’t documented – a trivial gap, but it highlights that some of these projects have minimal public marketing, focusing on internal use.

Suppositions

We suppose that BiJoe’s future will involve becoming more user-friendly in terms of creating and viewing analytics. Perhaps a visual query builder or a set of predefined dashboard templates will be introduced so that city staff can get insights without writing any code or even needing a developer. It’s likely that BiJoe, in conjunction with WCS-OLAP, will be expanded to cover data from other modules as well – for example, Chrono data (appointments) could be turned into an OLAP cube (e.g., facts of appointments with dimensions like service type, fulfilled or not, etc.), or Lingo’s payments could form a cube (payments fact with amount, date, status, payer category). If not already, we suspect those could come via analogous “*-OLAP” extensions for each domain. Another guess: BiJoe might integrate with external BI tools or standards – for instance, exposing a standard OLAP interface (like XMLA or an MDX query endpoint) so that third-party BI software could connect. However, given its framing as a Django app, it may remain a more self-contained solution. We also think that as more data is collected in Publik, the importance of BiJoe will grow; thus the maintainers might invest in optimizing query performance (maybe using materialized views or summary tables to speed up common queries). On the development side, a new coder might suppose that to add a new metric, they would define a new model or view in the star schema and then use BiJoe’s library to query it – future improvements might simplify this to configuration rather than coding. As for naming, “BIJoe” suggests possibly an acronym (Business Intelligence) combined with a play on GI Joe (the action figure) – it’s not a technical supposition, but indicates the team’s informal naming style. In conclusion, BiJoe is expected to continue as the analytical pillar of Publik, potentially incorporating more automation (regular reports, trend analysis) and broadening its data coverage, so that stakeholders can readily evaluate the performance and usage of their digital services within Publik.
