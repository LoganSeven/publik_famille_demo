Fargo (Document Management “Parapheur”)

Summary & Integration

Fargo is Publik’s document management module, described as the “porte-documents” (document briefcase) of the ecosystem[40]. In practical terms, it serves as an electronic document cabinet and possibly an electronic signature workflow (the term parapheur in French administration refers to a system for circulating and signing official documents). Fargo’s role in the ecosystem is to handle uploaded or generated documents that are part of user requests or internal processes. For example, when a citizen submits supporting documents through a form (attachments) or when an agent produces a PDF report or decision letter, Fargo would store these files and manage their lifecycle. It likely integrates with WCS (forms can send their file attachments to Fargo for storage), and possibly with the portal (Combo) to allow users or agents to download documents from their case files. Fargo might also interface with external archiving or signing tools, but within Publik it is the central repository for unstructured data (files) related to workflows. It ensures that documents are available to the right users: an agent can retrieve a document from Fargo via the back-office, or a user might get a copy of an official document via the portal, with Fargo delivering it. Integration points also include potential triggers in WCS workflows: e.g., at a certain step a PDF is generated and stored in Fargo, or a request is marked as needing a signature, which Fargo facilitates.

Philosophy, Design & Usage

The philosophy behind Fargo is to provide a unified document repository that plugs into digital workflows. Instead of scattering files on disk or embedding them in form data, Fargo offers a structured way to manage documents, likely including metadata, access control, and possibly versioning or validation. It’s designed to meet the needs of government processes where official documents must be handled securely and often signed or approved. The choice to separate Fargo as a module suggests a separation of concerns: WCS handles form logic, while Fargo handles document storage and processing. Fargo is presumably built with Django as well (consistent with the others), managing models for documents, folders, and perhaps signature status. Its usage involves both citizens and agents: a citizen might upload files through a form (those go into Fargo under the hood), and an agent might add or modify documents during case processing. There might be a user interface (maybe via Combo or the agent portal) that allows browsing a “dossier” of documents for a request. The design likely includes features such as permission control, ensuring only authorized personnel can see certain documents (for instance, an HR document vs a public document). Because it is referred to as a parapheur, Fargo possibly supports workflows like routing a document to multiple signatories or stamping a document when approved. However, given limited info, it seems the core design is a document store with some workflow hooks. In usage, an administrator would set up Fargo as part of the system so that every submission or generated output gets stored in a central location (often /var/lib/fargo on the server, based on typical deployment conventions). The idea is that all files are handled by Fargo service, simplifying backup and security (versus being spread out). Users interacting with Fargo do so indirectly: e.g., when they click “Download attachment” in the portal or when an agent clicks “Sign document”, behind the scenes Fargo is serving or updating that file.

Key Features & Functions

Fargo’s essential features likely include: document storage and retrieval, document organization, and possibly signature workflow support. As a storage engine, it handles saving files (perhaps in a database or filesystem with references). It might generate unique IDs or filenames, and ensure files are not lost or duplicated. As a “briefcase”, it might group related documents – for instance, all documents for a given WCS submission or for a certain case can be treated as a set. This could manifest as a folder or dossier concept in Fargo. Another key function is providing a link between documents and Publik entities: Fargo would have an API or signals such that when WCS workflow reaches a “signing” step, Fargo knows which document to present for signature to which agent. There is also likely a function for anonymization or archival: over time, documents might need to be archived or purged for privacy (some support might exist, akin to Chrono’s anonymize bookings for personal data – Fargo might have to anonymize or delete files after retention period, though this is speculative). Additionally, Fargo probably exposes download endpoints (ensuring authentication) so that Combo can offer direct links for users to get their files. Considering the e-signature aspect, Fargo might integrate with a signing tool or allow an agent to apply a signature stamp. At minimum, it would allow uploading a signed version of a file or storing a signature certificate. The initial release of Fargo (version 1.x[41] suggests it’s relatively new and basic) may focus on storage; essential functions would include creating document records, linking them to form submissions or other contexts, and providing CRUD (create, read, update, delete) operations via an admin or API. The “porte-documents” metaphor implies it holds documents in a way accessible to users – for example, a citizen might have a personal space listing the documents they’ve submitted (or permits they’ve received), implemented via Fargo linking with their account. In summary, Fargo’s primary function is to ensure that all files in Publik workflows are properly managed and accessible when needed, acting as the digital filing cabinet for the platform.

Unknowns or Gaps

Fargo is relatively less documented externally, so several aspects are unknown. For one, the full scope of “parapheur” functionality is unclear – does Fargo simply store documents, or does it also manage signing flows (like sending documents to officials for e-signature and tracking that)? The one-line description[42] doesn’t elaborate. If electronic signature is part of Fargo’s purpose, the details of how it’s implemented (what cryptographic tools or standards are used) are not public. Another unknown is the data model: the Chrono wiki references a generated model graph for agendas, but we have none for Fargo – we don’t know if Fargo uses a hierarchical folder model or just tags documents by case ID, etc. Also, it’s not documented how Fargo interacts with external systems – for instance, some cities use dedicated archival systems or external parapheurs; whether Fargo can export or sync with those is not stated. Performance and storage strategy (files on disk vs in database as BLOBs) is also not clear from available info. Additionally, how Fargo ensures security (encryption at rest? virus scanning of uploads?) is not described in the docs we saw – likely such considerations exist internally, but they’re not mentioned. For new developers, finding API or usage documentation for Fargo might be a challenge; it appears to rely on internal knowledge or code reading to fully grasp capabilities. Finally, Fargo’s interaction with multi-tenant setups is an open question: presumably each tenant’s documents are siloed, but the mechanism (separate file directories or tenant IDs) isn’t obvious without deeper docs.

Suppositions

Based on its intended purpose, we suppose that Fargo will evolve to include more robust parapheur features such as digital signing and document workflow. Possibly, upcoming commits might introduce integration with France’s national electronic signature mechanisms or adherence to archiving standards (since public sector document management often requires compliance to norms like PDF/A, timestamps, etc.). Another guess is that Fargo might eventually offer a user-facing document portal – for example, allowing users to see all their official documents (permits, certificates) in one place. This could tie into Publik’s goal of a unified user account experience. We also suppose that Fargo uses file storage on the server’s filesystem (with paths per tenant) and keeps metadata in the database – a standard approach, though not confirmed. Given that Fargo is at version 1.x, it is likely under active development; we anticipate new features such as bulk download, integration with scanning or OCR for incoming paper documents, or more administrative tools for document retention policies. For developers, an educated guess is that Fargo’s API is similar to other modules – possibly a REST API to list and fetch documents, and maybe signals for WCS to push files to it. If Fargo does not yet handle electronic signature internally, one might have to integrate it with a third-party signing service; however, since Entr’ouvert also has a project called Cryptic (for digital certificates)[43], it’s plausible that Fargo will leverage such components to offer in-house signing in the future. In conclusion, Fargo is expected to grow from a basic document repository into a more feature-rich document workflow system, aligning with the increasing digitalization of administrative paperwork.
