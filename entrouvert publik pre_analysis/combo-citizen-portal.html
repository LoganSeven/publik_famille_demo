Combo (Citizen Portal)

Summary & Integration

Combo is the content management system (CMS) serving as Publik’s front-office user portal and also the back-office agent portal[1]. It is a Django-based web application used to build and customize portals for citizens and staff[2]. Combo handles the website pages, templates, and theming, allowing each city or tenant to have a uniquely branded portal. It integrates tightly with the Authentic 2 identity provider for user accounts and single sign-on (SSO): the portal delegates authentication and personal data management to Authentic, ensuring secure login and user profile handling[3]. Through a system of pluggable content “cells”, Combo can display dynamic information from other Publik modules. For example, when a user logs in, a cell can show the list of that user’s ongoing requests by querying the WCS service in real time[4]. This design means Combo acts as the face of the ecosystem, aggregating data from WCS (for e-forms), Lingo (for payments), Chrono (for appointments), etc., into a unified portal experience.

Philosophy, Design & Usage

The philosophy behind Combo is to provide a simplified CMS tailored for e-government portals. It uses Django’s template framework to enable fine-grained control over page layouts and style[5], so that cities can implement their web design strategy within the portal. Each page is composed of cells (content widgets), which can be static content or live data from external systems, allowing flexible integration of third-party or backend information[6][4]. This approach promotes a modular design – new cells or features can be added via plugins (indeed, several Combo plugins exist for specific needs). Coding choices favor reuse of Django components, keeping Combo relatively lightweight as a CMS. The user experience is kept simple by design – navigation and ergonomics are meant to be intuitive for citizens[7]. In usage, administrators (via the agent portal, which is essentially the same Combo application in back-office mode) can create and edit pages or place cells without deep technical skills. The integration choices (templating system, plugin architecture, and connectors) reflect a goal of openness: Combo doesn’t reinvent content storage but rather focuses on presenting information and interfacing with other modules through standard web techniques.

Key Features & Functions

Combo’s core functions include managing portal pages, menus, and content templates. It provides authentication hooks to work with SSO (users can log in via Authentic 2, or FranceConnect federated login)[3]. It also includes a role-based view: the same portal can serve different audiences (citizens vs agents) by presenting different sections or requiring login. A notable feature is the cell system – cells are configurable components on a page that can display, for instance, news feeds, maps, or personalized data. For personalized data, Combo calls the relevant backend module via its API at page render time (e.g. fetching a user’s submitted forms from WCS)[4]. Combo also supports custom Django template tags (the project publik-django-templatetags provides shared filters and tags for Publik apps[8][9]). In terms of extensibility, Combo’s repository includes plugin modules (e.g. combo-plugin-gnm, combo-plugin-nanterre as listed in the package index[10]) which add localized functionalities or integrations. This modular structure is an essential function: new features can be rolled out as separate plugins rather than altering core code. Overall, Combo’s initial essential capability is serving as a portal framework – it renders pages, interfaces with user authentication, and pulls in content from various sources on the fly.

Unknowns or Gaps

Specific implementation details of Combo’s internal architecture are not extensively documented publicly. For example, the precise mechanism for the front-office/back-office distinction is not explicitly described in available sources – it is likely done via Django settings or separate site instances, but this is not confirmed in documentation. Another unknown is the full extent of Combo’s plugin API for developers (i.e. how one writes new “cells” or extensions); this likely exists in the code but is not covered in high-level docs. It’s also not explicitly stated how Combo handles content versioning or publishing workflows – presumably it’s a simple draft/publish model, but documentation is sparse. Additionally, while we know Combo relies on Authentic for user data, it’s not clear if Combo stores any user profile info locally or if absolutely all profile management is offloaded to Authentic (the assumption is the latter, but details are not found in the docs). These areas would require reading the source to fully understand. Finally, performance considerations (caching of cell data, etc.) are not discussed in the docs – it’s unknown if Combo caches external data or always fetches fresh data for each page view.

Suppositions

Based on its role, one can suppose that Combo’s development is geared toward increasing flexibility and integration. For instance, recent commits have likely focused on improvements like new cell types or better error handling for content (indeed, an example fix in early 2025 addressed file name length issues for page “picture” attachments[11], showing ongoing refinement). It can be supposed that future directions might include more built-in connectors or visualization cells (there have been mentions of dataviz features in development tickets, suggesting possible chart or dashboard cells). Another supposition is that Combo will continue to serve as the layout engine while heavy business logic lives in other modules – meaning the team likely keeps Combo’s core simple and offloads new functional requirements to specialized services or plugins. Given the trend towards multi-tenant use, we also surmise that Combo supports multi-tenancy via Hobo (e.g., one Combo instance can host multiple portals for different tenants, separated by URL and branding), though the exact mechanism is abstracted by Hobo. In summary, Combo is expected to remain the adaptable front-end that evolves mainly through extensions and theming capabilities, rather than drastic architectural changes.
