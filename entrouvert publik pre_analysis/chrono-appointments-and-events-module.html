Chrono (Appointments & Events Module)

Summary & Integration

Chrono is Publik’s appointment scheduling module, used for managing rendez-vous and event bookings[44]. It provides the functionality for citizens to book time slots for services (e.g. appointments at city hall, meetings with an official, reserving a place in an event) and for agents to administrate these schedules. In the context of Publik, Chrono works closely with other components: for instance, an online procedure in WCS may require the user to choose an appointment slot (Chrono supplies the available slots), or the Combo portal might offer a “Book an appointment” service powered by Chrono. Chrono essentially acts as the central calendar/agenda system for Publik, ensuring that bookings are coordinated and avoid double-booking. It can integrate with external calendar systems if needed (though primarily it maintains its own data). Chrono’s data (agendas, slots, bookings) can also feed into BiJoe for statistics (like number of appointments, usage patterns), and Chrono may use Passerelle if it needs to fetch information (for instance, confirming an appointment in an external agenda if one exists). However, primarily Chrono is an internal service that other Publik modules call. One integration example: a WCS form for scheduling a meeting might call Chrono’s API to create a booking when the form is submitted, or conversely, Chrono might trigger a WCS workflow when an appointment is made. Chrono ensures that the citizen receives a confirmation (likely via email, possibly through Combo or directly).

Philosophy, Design & Usage

The philosophy of Chrono is to provide a structured scheduling system that fits into e-government workflows. Many public services require managing queues and appointments (for passports, permits, etc.), so Chrono is designed to handle these scenarios out-of-the-box. It likely uses a model of calendars (agendas), time slots, and bookings (reservations). The design is that an administrator can define one or more agendas (for example, one agenda per service or per location), set up opening hours or available time slots, and then the public can book those slots. Chrono’s name suggests a focus on time; accordingly, its design probably deals with time zones, recurring availability, and possibly capacity management (like how many people per slot). Chrono is implemented as a Django app (consistent style), and the documentation hints at an API and even data model diagrams for agendas[45]. It likely uses Django models for events and reservations, and may also incorporate the django-tenant-schemas via Hobo for multi-tenancy, so each tenant’s appointments are isolated. In terms of usage, staff users can use Chrono’s back-office interface (possibly integrated in the agent portal) to create schedules, manually add or modify bookings, or block out times. Citizens use the front-office (via Combo) to see available appointment slots and book them – Chrono provides that data and enforces rules (like preventing double booking the same slot). The design choices include providing a management command for tasks like anonymization of old bookings (to comply with data protection)[46], indicating an awareness of privacy in the design. Chrono is also built to handle events beyond one-on-one appointments – possibly small events or multi-person slots, given it mentions “évènements” (events) along with appointments[44]. This could mean it can manage sign-ups for limited-capacity events in addition to standard appointments. All these suggest Chrono’s usage scenario is broad: any scheduling need within Publik should be served by Chrono rather than using an external calendar.

Key Features & Functions

Chrono’s key features include agenda management, slot definition, booking workflow, and possibly notifications. Administrators can define one or multiple agendas (calendars) – each might correspond to a specific service desk or agent or type of appointment. For each agenda, they can define available time slots (e.g., Monday 9–12 every 15 minutes). Chrono likely has a recurring schedule feature to avoid manually inputting every slot. Citizens can then see these slots and reserve one. The booking function ensures that when a slot is taken, it’s marked as booked and no one else can take it. Chrono probably collects user details during booking (either linked from their account or via form fields) and may issue confirmation emails automatically. Another function is booking modification/cancellation – users might be able to cancel or reschedule via a link or by contacting an agent, which Chrono would handle by freeing up slots. On the back-end, Chrono provides a view for agents to see the schedule (maybe a calendar view) and the list of bookings, with options to add walk-in appointments or block times for meetings. It also has an API that other modules can call; indeed, there is mention of an API documentation in the Chrono wiki[47], suggesting one can programmatically create or query bookings. Chrono likely supports exporting calendars as iCal/ICS – possibly an agent could export an agenda to integrate with their personal calendar software (the WCS code had references to .ics generation for events[48], which might relate to Chrono’s data). Additionally, Chrono includes maintenance commands; for example, an anonymize_bookings command is present to wipe personal data from past appointments[46], which is an important function for GDPR compliance. This shows Chrono’s awareness of data lifecycle. In summary, Chrono’s initial essential functions are to allow creation of schedules, booking of time slots by users, management of those bookings by staff, and to ensure no conflicts in scheduling – with supplementary features like notifications and data management to round it out.

Unknowns or Gaps

Certain aspects of Chrono remain unclear from the available info. For instance, how does Chrono handle overbooking or walk-ins? It’s not specified if multiple bookings per slot are allowed (presumably not unless configured as a multi-capacity event) or if Chrono can manage wait-lists (the Chrono wiki alludes to “gestion des files d’attente” – waiting queue management – as a topic of reflection[49], implying that feature might be undeveloped or experimental). Also unknown is whether Chrono has any integration with external calendar systems (like syncing with an Outlook/Exchange or Google Calendar, which some public offices might use). There’s no explicit mention, so it might be standalone. Another gap is the notifications mechanism – Chrono presumably needs to send confirmation or reminder emails; whether it relies on Publik’s general email configuration or has its own scheduling for reminders is not detailed publicly. The extent of the API is also not fully known: we see references to management commands and an API doc, but not the specifics of endpoints or features (like can you search for bookings by name, etc.). Additionally, how Chrono interacts with identity management is a bit unclear – does it require a user account to book an appointment? Likely yes for traceability, but maybe it also allows anonymous bookings with contact info. Such details are not visible in the brief descriptions. Finally, performance under heavy use (e.g., a very popular booking that opens hundreds of slots and many users try to grab them at once) is not described – whether Chrono implements any queue or atomic locking beyond database transactions (to avoid race conditions) is unknown. These are the kind of details that might only be found by examining code or detailed admin guides.

Suppositions

We suppose that Chrono will be enhanced particularly in the area of queue management and capacity handling. The presence of discussions about waiting queues[49] suggests future versions might allow citizens to join a waitlist if no slots are available, or manage a queue of people for first-come first-served scenarios (for example, virtual queue numbers). Another likely development is improved notifications – possibly SMS reminders or integrations with calendar invites (sending an .ics file in confirmation emails). Given Chrono’s nature, we can guess that it might integrate with Lingo or payments if any appointments require a fee (though not a common scenario, but imagine booking a facility that requires payment – Chrono could generate a payment link via Lingo). We also anticipate Chrono’s API becoming more RESTful and comprehensive, to allow cities to embed calendars in their own interfaces if needed. A supposition on design: Chrono probably uses Django’s scheduling capabilities or custom logic to handle recurring availability; if not, that could be an area of future improvement (making it easier to set complex schedules, holidays, etc.). We also guess that Chrono may eventually incorporate resource management – e.g., ensuring that if a given agent or room is booked, it’s reflected across multiple agendas (prevent double-booking the same person in two places). For multi-tenant setups, each tenant likely has separate agendas, and Hobo ensures queries go to the right schema – this we assume is handled behind the scenes. Lastly, given Chrono’s data (appointments) are sensitive, we suppose features like data retention policies (auto-delete personal data after X time) will continue to be important, as evidenced by the anonymization command. In summary, Chrono is expected to mature into a more full-featured scheduling system, possibly incorporating user feedback like managing no-shows or integrating with check-in systems, all while remaining an integral part of the Publik suite rather than a standalone calendar.
