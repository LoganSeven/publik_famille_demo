WCS-OLAP Extension

Summary & Integration

WCS-OLAP is an extension module that connects WCS (the form/workflow engine) to the OLAP system (BiJoe) for business intelligence. Its purpose is to export or transform WCS data into OLAP cubes suitable for analysis[52]. Essentially, it extracts data from WCS’s operational database – such as submissions, their attributes, and statuses – and structures it into the star schema that BiJoe expects. In the Publik ecosystem, WCS-OLAP is the bridge between the live e-service data and the reporting/analytics layer. It likely runs as a batch job or set of routines that periodically take snapshots of WCS data and load them into BiJoe’s database (or into a schema dedicated to analytics). Integration-wise, WCS-OLAP interacts with both WCS and BiJoe: it reads WCS models (perhaps via direct database access or through an API) and populates the corresponding BiJoe models or tables. It may also integrate with Hobo’s scheduling (for running periodic updates). Importantly, this module means that end-users (administrators) don’t have to manually create their own data extracts – it automates the flow of data into BI. This extension is specific to WCS data, so each Publik instance that wants form analytics would have WCS-OLAP installed alongside WCS and BiJoe.

Philosophy, Design & Usage

The philosophy of WCS-OLAP is automation and consistency in data analysis. By providing a predefined way to convert WCS data to OLAP, it ensures that all installations can get a standard set of metrics without custom development. The design likely involves mapping core WCS entities to facts and dimensions. For example, each form submission could become a fact record with measures like “count=1” (for counting submissions) or “processing_time” (for average duration calculations), and dimensions like date submitted, form type, current status, etc. The usage of WCS-OLAP is probably largely behind the scenes: an admin might install or enable it, and from then on it updates the data warehouse. Some manual trigger might exist (like a management command wcs-olap export to force an update, or it could hook into WCS events in real-time to update incrementally). Given that the description is “export des données pour construction de cubes OLAP”[52], it suggests a focus on exporting data. Possibly the design is such that on a schedule, it dumps WCS data into CSVs or directly into the BI database. Another design consideration is data volume – WCS might accumulate a lot of submissions, so WCS-OLAP might do incremental updates (only new or changed records) rather than full reloads each time. From a coding perspective, WCS-OLAP is probably a Django app that knows about WCS’s models (hence it might depend on WCS’s code or share database access). Usage from a developer’s standpoint might involve configuring which forms or which fields to include in the OLAP export, but it’s likely that by default it exports all relevant data (perhaps excluding extremely detailed form fields, focusing on higher-level metrics). The user (analyst) doesn’t use WCS-OLAP directly; they see the results in BiJoe queries or dashboards.

Key Features & Functions

The primary function of WCS-OLAP is to transform WCS data into a star schema for analysis. Key features likely include: selecting the data to export (e.g., all form submissions, perhaps aggregated by form definition), creating dimension tables (like a dimension for dates – e.g., submission date, completion date, etc., and a dimension for form types, and maybe for organizational units if forms are categorized), and populating one or several fact tables (such as “FormSubmissionFact” containing one record per submission with various metrics). It might also calculate some metrics during export – for instance, computing the time difference between submission and resolution of a request, so that it can be used in average processing time reports. Another feature is ensuring referential integrity and updates: if something changes in WCS (like a request’s status is updated to “done”), WCS-OLAP should eventually update the fact record to reflect that (maybe storing the final status or closing date). It may handle historical snapshots too (keeping track of counts over time). Given version 1.64[32], WCS-OLAP is mature enough to have refined these features. Possibly, it includes management commands or a daemon that performs the extraction. There might also be configurations to exclude certain form fields or include custom metrics. Overall, WCS-OLAP’s essential function is the ETL (Extract, Transform, Load) of WCS data into BI form. Without it, an administrator would have to manually script data dumps for analysis; this module provides a consistent, repeatable process.

Unknowns or Gaps

Being a behind-the-scenes module, WCS-OLAP is not well-documented in public. We don’t have details on how it is triggered – whether it’s real-time, on form submission events, or a scheduled batch. It’s unknown how it handles changes (for example, if an existing submission’s data is modified or if a workflow loops through statuses, does the OLAP data update accordingly or just capture the final state?). The specific schema it creates is also not described anywhere publicly – one would need to inspect the database after running it. Another unknown is if WCS-OLAP can be configured easily for custom form fields: by default, it might focus on general properties (like form name, status, dates), but if a city wants to analyze a specific field (e.g., how many requests with “urgent” flag), it’s unclear if that’s directly supported or requires custom work. Additionally, how WCS-OLAP deals with multi-tenant is an open question: presumably each tenant’s data is separate in the BI schema as well (maybe using separate schemas or prefixed tables for each tenant). The performance impact on the WCS database when exporting large volumes is also not documented – hopefully minimal, but not stated. Lastly, since WCS-OLAP is specific, we don’t know if similar modules exist for Chrono or Lingo (none are listed by name), which might indicate that WCS-OLAP is a one-off and other modules’ data might be integrated into the same OLAP cubes via this one extension or not at all. This is unclear.

Suppositions

We suppose that WCS-OLAP will expand or be accompanied by similar OLAP exporters for other data sets, or it might already be evolving to cover more than just WCS. Perhaps the generic name “OLAP / Business Intelligence pour Publik”[52] on Redmine hints that WCS-OLAP is part of a broader effort, and in the future it could incorporate Chrono (appointments) data exports or Lingo (payments) data. It’s likely that WCS-OLAP’s maintainers will update it as WCS evolves – for instance, if WCS gets new features (like new types of workflow data), the OLAP export would adjust to include those in analysis. We guess that WCS-OLAP runs periodically (nightly or hourly), as real-time OLAP might not be necessary; this means future versions might introduce configuration of the schedule or on-demand refresh. Another supposition is that as BiJoe gets more front-end capabilities, WCS-OLAP might also include some predefined reports or cubes. It may create not just raw fact tables but also some summary tables for common metrics (to speed up dashboards). For example, it might pre-aggregate counts per month to make trend charts faster – an optimization we suspect might come if not already present. We also assume that deploying WCS-OLAP requires aligning versions with WCS and BiJoe; thus, developers need to ensure compatibility, which might be noted in release notes (though not explicitly seen by us, likely internal knowledge). In terms of usage, we can imagine future enhancements like a UI showing when the last data export ran and how many records were processed, to give admins confidence their BI data is up to date – this is speculative, but a logical improvement. All in all, WCS-OLAP is an enabling tool that likely stays mostly in the background. We anticipate it will continue to be maintained as an important piece for any Publik installation that values metrics and will adapt as new analytics needs arise (for instance, measuring form abandonment rates or durations might be a next step to incorporate into the OLAP dataset).
